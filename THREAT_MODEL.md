# Trace threat model

## Overview

Trace is a static browser application with a Rust/WebAssembly core. A person
supplies an iPhone sysdiagnose archive; the engine streams gzip/tar content,
parses four primary phone diagnostic surfaces plus explicitly labeled
paired-device diagnostics, applies reviewed public spyware indicators, and
emits a schema-versioned JSON report. There is no application upload endpoint.
The primary runtime surfaces are the browser application under `web/`, the
scanner and parsers under `crates/trace-core/`, the service worker, the bundled
indicator snapshots, and the CI/deployment path that produces the served
application.

The security goal is not “prove the phone is clean.” It is to produce a
truthfully scoped, locally computed result without exposing the person's
archive, and never turn missing, malformed, truncated, over-limit, or
unrecognized evidence into a reassuring clear verdict.

This model covers repository and operational risks. It does not establish the
operator's legal identity, funding, governance, continuity, or external-audit
status; those facts are not documented by the repository and remain trust
questions for an adopting organization.

### Assets and security objectives

- **Person and case safety:** using Trace must not silently expose a person who
  may already be under surveillance.
- **Sysdiagnose confidentiality and integrity:** the archive can contain
  sensitive data beyond what Trace reads. Its bytes must remain local to the
  intended scanner and remain identifiable after transfer.
- **Verdict integrity:** matches, suspicious anomalies, parser degradation,
  missing surfaces, and scan limits must not be suppressed or softened.
- **Report integrity and interpretability:** a responder must be able to bind a
  report to exact archive bytes, identify the source revision and indicator
  snapshots, understand coverage, and reproduce security semantics.
- **Indicator integrity:** only reviewed snapshots should affect matching, and
  the exact text used must be identifiable.
- **Application and deployment integrity:** the code served to a browser should
  correspond to reviewed source and should not be replaceable by a hostile
  dependency, host, cache, or deployment path.
- **Availability:** a hostile archive should not cause unbounded browser memory,
  CPU, DOM, or decompression use, and a failed scan must fail honestly.

## Threat Model, Trust Boundaries, and Assumptions

### Actors and adversaries

- The person capturing and scanning their phone, who may be nontechnical and at
  physical or digital risk.
- A responder who receives a report and possibly an archive, and who must treat
  both as untrusted, sensitive evidence.
- An adversary with control of or visibility into the iPhone, Apple account,
  local network, DNS resolver, shared computer, synchronized storage, browser,
  browser extensions, or endpoint-monitoring software.
- A malicious archive author able to control gzip, tar headers and metadata,
  file sizes and paths, diagnostic text/JSON, and unified-log payloads.
- A network, hosting, service-worker, repository, dependency, CI, or deployment
  attacker attempting to replace the application or observe use.
- A threat-intelligence upstream that is unavailable, compromised, malformed,
  stale, or legitimately changed.
- Repository maintainers and deployment operators, who control reviewed
  snapshots, source, release tags, and production configuration. Their legal
  identity and governance are outside what this repository establishes.

### Trust boundaries

1. **Phone to diagnostic archive.** Trace begins with data generated by the
   device under investigation. The device determines what enters its
   sysdiagnose before Trace can hash or parse it.
2. **Archive transfer and storage.** AirDrop, cable transfer, iCloud Drive,
   `icloud.com`, local disks, downloads, backups, and case-management systems
   can create additional copies, access logs, notifications, or synchronized
   artifacts.
3. **Network and host to browser.** The browser obtains HTML, JavaScript,
   WebAssembly, indicator snapshots, headers, and service-worker state from the
   hosting path. DNS, TLS connection metadata, IP addresses, and host access
   logs are outside the in-tab scanner boundary.
4. **Untrusted archive to parser.** Attacker-controlled compressed bytes cross
   into gzip/tar parsing, diagnostic parsers, the upstream unified-log parser,
   matching, report serialization, and eventually DOM rendering.
5. **Threat intelligence to reviewed snapshot.** Public upstream STIX is
   untrusted input to the review/update process. At runtime, the reviewed
   bundled snapshots affect matching; live upstream fetches affect only an
   advisory different-content notice that does not establish recency.
6. **Engine report to browser and filesystem.** Rust owns the report envelope
   and verdict, but JavaScript displays and exports it. Once downloaded or
   shared, the unsigned JSON and derived readable HTML can be edited and can
   expose sensitive metadata.
7. **Source, tag, build, and deployment.** A source commit and version tag are
   not the same thing as a served binary. `tool.build_commit` records the
   revision string injected by the build path. Official CI and deployment
   inject the checked-out commit; a custom build or edited report can claim a
   different value. The repository does not currently provide third-party-
   checkable byte-for-byte deployed-WASM reproducibility.
8. **Report handoff to responder.** A responder must independently preserve and
   hash both files, match the archive to `source_file.sha256`, validate the
   schema locally, and re-scan when the decision warrants it.

### Input control

- **Attacker-controlled:** archive bytes; gzip/tar structure, paths, sizes, and
  metadata; all retained diagnostic content; any report received by a
  responder; network behavior; and, after a device compromise, the diagnostic
  evidence itself.
- **Operator-controlled:** the deployed static files, hosting configuration,
  release process, reviewed indicator selection, and retention of redirects and
  service-worker behavior.
- **Developer-controlled:** scanner code, parser dependencies, schemas, tests,
  resource limits, build tooling, and release tags.
- **Upstream-controlled but verdict-isolated at runtime:** live public indicator
  URLs. Their response can change the advisory `upstream` comparison status, but
  not the indicators used for matching. Reviewed snapshot content becomes
  developer/operator-controlled when committed.
- **Producer-supplied descriptive metadata:** source filename and browser
  producer mode. These values must not affect the verdict. The engine computes
  the archive size and hash from every byte received, and hashes the exact
  indicator text used.

### Security invariants

- Intended Trace code sends neither archive bytes nor report contents to an
  application endpoint. Loading and upstream-comparison requests must never
  include scan data.
- The Rust engine is the only verdict authority. Browser rendering must not
  re-derive or soften it.
- A clear verdict requires recognizable evidence, loaded applicable
  indicators, no indicator-match or suspicious finding, no parser degradation,
  and no verdict-relevant resource or integrity limit. Bounded evidence
  sampling that cannot hide an identity remains separately disclosed. Missing
  surface types remain explicitly absent rather than implicitly examined.
- Every archive byte received by the engine contributes to
  `source_file.sha256`; the exact loaded text for each indicator set contributes
  to its `indicator_provenance[].sha256`.
- An indicator match or suspicious anomaly must remain visible even when another
  part of the scan is incomplete. Findings caps must retain higher-severity
  evidence and disclose the cap.
- Corrupt, truncated, oversized, unsupported, or adversarial inputs must fail
  as an error, `invalid`, `inconclusive`, or a non-clear finding-bearing report,
  never as a false clear caused by the failure.
- All report producers must emit the same versioned contract. Schema validity
  must not be mistaken for report authenticity.

### Assumptions and explicit limits

- The browser, operating system, and cryptographic implementation are trusted.
  A compromised browser, extension, endpoint, or host can read files selected
  by the person and invalidate the local-processing guarantee.
- The person receives the intended Trace application. A hostile initial page or
  persisted service worker can behave differently from reviewed source; going
  offline after loading does not repair an already compromised application.
- SHA-256 collision resistance holds. A hash identifies bytes, not their origin
  or truthfulness.
- Public threat intelligence is necessarily incomplete and delayed. New,
  private, removed, or intentionally absent traces are not detectable merely by
  parsing correctly.
- A sufficiently compromised iPhone can lie in its own sysdiagnose, suppress
  artifacts, or observe capture. This is an acquisition limit, not a condition
  Trace can fix after the fact.
- Absence of a supported surface can be normal. `assurance.complete` describes
  processing of available input, not comprehensive device coverage.
- No real infected sysdiagnose is part of the public validation corpus; current
  infected fixtures are synthetic from published patterns. Clean-corpus results
  do not establish a population-wide false-positive rate.

## Attack Surface, Mitigations, and Attacker Stories

### Site visit and network observability

An observer can learn that the person resolved or connected to the distinctive
Trace hostname through DNS, TLS/SNI or destination-IP metadata, browser or
gateway logs, and hosting-edge records. Cloudflare can add network-error
reporting headers for delivery failures. The browser also attempts public
upstream indicator requests for an advisory difference check. None of this
should carry the archive or report, but the fact of use may itself be sensitive.

Loading the page before disconnecting and scanning offline reduces scan-time
network exposure and demonstrates the lack of an upload dependency. It does not
hide the initial visit. A person facing coercive control or targeted
surveillance may need a different network and device rather than instructions
to “just use private browsing.”

### Capture on a compromised phone

The volume/side-button gesture, diagnostic generation, system activity, and
resulting file exist on the possibly compromised phone. A capable implant or
person with device/account access may observe the capture, omit evidence,
falsify diagnostic content, or delete the archive. Trace hashes only the bytes
it receives; it cannot attest to the acquisition process. A negative result
therefore cannot override a credible external risk signal.

### Shared devices, browser history, and local artifacts

The URL, downloads, recent-file lists, search indexes, synced browser history,
cloud backup, security software, and local monitoring can expose that Trace was
used or reveal the report/archive. Private-browsing mode is not a complete
mitigation because downloaded and operating-system-level artifacts remain.
Responders should plan access, transfer, retention, and deletion around the
person's adversary model rather than make a universal “safe device” promise.

### Windows and iCloud transfer trade-off

The current Windows path can move the archive through iCloud Drive and
`icloud.com`. This can expose the archive or account activity to anyone with
Apple-account credentials, a trusted/synchronized device, notification access,
or cloud-provider/legal access. iCloud may be the only practical route for some
people; the risk is a disclosed trade-off, not a categorical prohibition. When
account compromise is plausible, choose a transfer route with a specialist
before generating another observable copy.

### Malicious-archive denial of service

The archive is fully attacker-controlled. Relevant classes include gzip bombs,
huge tar members, metadata floods, integer edge cases, corrupt checksums,
truncation, parser-complexity attacks, finding floods, and tracev3 metadata that
drives large allocations.

Existing controls in `crates/trace-core/src/tar_stream.rs` and
`crates/trace-core/src/report.rs` include streaming rather than materializing
the full archive, a 32 MiB per-file cap, 192 MiB aggregate retained-file cap,
4,096 retained artifacts, 1,000,000 walked headers, an 8 GiB decompressed-byte
budget, a 5,000-finding cap, checked/saturating arithmetic, tar checksum
validation, and explicit incomplete-scan reporting. Higher-severity findings
can evict lower-severity entries at the cap.

Before bytes reach the upstream unified-log parser, Trace validates complete
tracev3 framing plus catalog offsets, nested counts, and exact declared-body
consumption. It then exposes only catalog frames to that parser. Message
chunksets never cross the parser boundary, so attacker-controlled compressed
size declarations cannot trigger LZ4 allocation or retained firehose parsing.
These catalog checks sit inside the outer archive and retained-member limits
above.

Unified-log identity and path budgets are verdict-relevant because exhausting
them can hide a matchable binary. PID retention is different: catalogs can
legitimately contain thousands of PID observations for one long-lived binary,
while matching uses its already-retained UUID and canonical path. Per-process
and aggregate PID sample limits therefore remain bounded and disclosed as
evidence truncation, but do not set the identity cap or make a complete surface
inconclusive.

The controls do not prove bounded aggregate CPU use or browser-tab
responsiveness for every hostile archive. An attacker can still supply many
individually valid files that consume repeated parser work within the outer
budgets. Process-candidate retention is capped at 10,000 per `.ips` file, and a
hit makes that artifact partial, but a large diagnostic JSON structure can
still amplify temporary parser state before reduction. The expected security
consequence is availability loss and an error or incomplete scan, never a
trustworthy negative result. Repeated failure is itself a reason to preserve
the file and escalate.

### Parser and verdict-integrity attacks

An adversary may craft recognizable-looking artifacts that are empty,
partially parsed, schema-drifted, or arranged after a corrupt/over-limit archive
section in order to hide a known match. The central engine collects parser
health, missing surfaces, checksums, truncation and cap state before choosing a
verdict. `match` and `suspicious` take precedence over incomplete status, while
a no-finding scan with degradation becomes `inconclusive`. Regression tests and
property tests cover hostile bytes and vacuous-clear cases, but new parser
formats remain a continuing false-negative risk.

The observed iOS stackshot format has one narrowly recognized exception: a
type-1 transition row with the exact five-field task shape, an empty procname,
at least one thread, and every thread's state containing both `TH_TERMINATE` and
`TH_TERMINATE2`. Trace discloses that anonymous tombstone as a `note`, never
uses it as an IOC candidate or checked-process count, and refuses transition-only
inventories as process-bearing coverage. Alternate identity fields, unknown
transition types, noncanonical PIDs, and arbitrary blank names remain partial.
This is a format policy, not proof that a compromised phone reported every
process truthfully; device-side omission remains the acquisition limit above.

### Indicator substitution, staleness, and scope

The official browser's runtime matching uses committed, reviewed STIX
snapshots. The engine hashes the exact loaded text into
`indicator_provenance`; live upstream content is parsed only to detect a
different plausible file and never replaces the snapshot. A difference is not
proof that the upstream file is newer or safer. The browser rejects a bundled
set below its reviewed floor, and CI requires exact reviewed
indicator/applicable counts plus one malware object, so count regression or
inflation cannot ship without an explicit reviewed metadata change. The native
harness can be supplied explicit STIX paths; its provenance binds the report to
the exact supplied text rather than asserting that the files were reviewed.

Repository or review compromise can still alter a snapshot, and correct
snapshots can still be stale or incomplete. Most domain, URL, email, browsing,
message, network-usage, installed-app, configuration-profile, and arbitrary
filesystem evidence is outside the sysdiagnose surfaces Trace reads. Reports
must continue to disclose both per-scan coverage and public-intelligence lag.

### Hosting, dependency, build, and service-worker compromise

A hostile served JavaScript or WebAssembly bundle can read a file after the
person selects it, falsify a verdict, or transmit data despite the intended
architecture. A malicious or stale service worker can persist across visits.
Strict CSP, same-origin opener isolation (COOP), nosniff/referrer headers, a
framework-free static site, offline behavior, pinned toolchain/deployment
components, CI tests, and commit-bearing reports reduce risk but do not
independently prove that the served bytes match public source.

In the official CI and deployment paths, `tool.build_commit` distinguishes
builds of different checked-out commits even when the package version is
unchanged. In a received report it remains an unsigned claim. Tags map releases
to source, and deployment checks can confirm a served commit marker. Neither
feature is a signature or a byte-for-byte reproducible-build attestation. An
adopting organization that requires this assurance needs an independently
verified build/deployment path.

### Report tampering and reproducibility

The exported JSON is unsigned. An attacker can edit `verdict`, `findings`,
`source_file.sha256`, build metadata, or coverage while preserving schema
validity. Computing the report's SHA-256 at intake detects later change only; it
does not authenticate the report as received. Matching the independently
calculated archive SHA-256 to `source_file.sha256` shows only that the supplied
archive matches the hash value currently written in the report; it does not
prove Trace scanned those bytes or that they came from the claimed phone.

The practical consistency check is semantic reproduction: preserve both files,
treat the recorded revision as an untrusted claim, review and build it only in
a disposable no-credentials environment, disconnect before introducing an
archive copy, confirm the indicator hashes, scan the exact archive, and compare
verdict, findings, limits, assurance, coverage, artifacts, and provenance.
Time, duration, browser producer, filename, and the advisory upstream comparison
can legitimately differ. This detects ordinary report modification but does not
provide a formal chain of custody, signed origin, acquisition attestation, or
deployed-binary reproducibility. The full responder procedure is in
[`HELPLINE.md`](HELPLINE.md).

### Report and archive confidentiality

The JSON report excludes the archive itself but can contain device metadata,
diagnostic paths/timestamps, process names/paths, and finding evidence. The
derived HTML export omits selected fields by default but still carries the
verdict, finding summaries (which can name relevant processes or paths),
archive hash, coverage, limits, and indicator provenance. The archive can
contain substantially more sensitive sysdiagnose data. Rendering, copying,
uploading to a public validator, attaching to an issue, ordinary email, cloud
synchronization, and long-lived case storage all create disclosure paths.
Minimize collection, access, copies, and retention, and use approved
confidential channels.

### Less relevant or out-of-scope attacker stories

- Trace has no end-user application accounts, sessions, tenants, scan API, or
  application database, so in-app authentication, CSRF, and tenant-isolation
  classes do not apply. Compromise of a person's Apple account or of maintainer
  GitHub/Cloudflare accounts remains in scope at the acquisition and deployment
  trust boundaries described above.
- A report that public threat intelligence inherently lags reality is not by
  itself a repository vulnerability. Runtime influence from unreviewed live
  data, a reviewed snapshot falling below the enforced floor, incomplete
  snapshot loading that can still produce `clear`, or incorrect upstream-
  comparison and provenance reporting can violate verdict integrity and remain
  in scope.
- The phone deliberately lying in its own diagnostics is an explicit evidence
  acquisition limitation. Trace remains responsible for describing it and for
  not overstating a negative result.
- Local filesystem access by a responder who was deliberately given an archive
  is expected. Unnecessary copying, insecure retention, or access beyond that
  purpose is an operational failure rather than a browser-parser vulnerability.

## Severity Calibration (Critical, High, Medium, Low)

Severity depends on realistic reachability and on consequences for people who
may be at risk.

### Critical

- Served or repository-controlled code sends selected archive bytes, report
  contents, or equivalent sensitive evidence to an unauthorized party for
  ordinary users.
- A reachable parser, verdict, or findings-retention flaw reliably converts an
  archive containing a known applicable indicator match into `clear` without
  disclosing the failure.
- A production/dependency/service-worker compromise persistently substitutes a
  scanner that exfiltrates files or fabricates reassuring results at scale.

These violate the central confidentiality or safety invariant and can directly
endanger users.

### High

- A crafted archive suppresses a known match, misbinds a report to different
  archive bytes, or produces materially false coverage/assurance without an
  obvious failure state.
- Report or build-provenance logic presents an unrecorded or different scanner
  revision as the recorded commit in a way that defeats a responder's re-scan.
- A realistic same-origin injection exposes sensitive report/archive data after
  file selection, even if exploitation requires an additional hosting or cache
  condition.

### Medium

- A malicious archive causes repeatable browser memory exhaustion, excessive
  CPU use, or tab failure without code execution or false-clear output.
- A parser rejects a legitimate, common sysdiagnose format and produces an
  explicit `inconclusive` result, impairing availability but not silently
  reassuring the user.
- A privacy control leaks use metadata or report metadata only under a meaningful
  additional precondition, such as an already monitored shared account, while
  archive contents remain local.

For a highly targeted person, operational metadata exposure can warrant a
higher case-specific response even if the software-class severity is medium.

### Low

- A UI, accessibility, wording, or provenance-display defect causes limited
  confusion but leaves the engine verdict, exported report, archive hash,
  coverage, and escalation path intact.
- A local-only development or test issue cannot reach the shipped scanner or
  alter release artifacts.
- Benign metadata differences prevent byte-for-byte report comparison while the
  documented security-relevant semantic fields remain identical.
